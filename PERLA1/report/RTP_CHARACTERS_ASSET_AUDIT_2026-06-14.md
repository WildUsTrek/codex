## Why RTP Is Heavy

1. Source packages are nested zip files. Browsers cannot use individual files inside those zips as normal static assets without runtime decompression.
2. Standard packages include both `spritesheet.png` atlases and `frames/*.png`, which duplicates the same animation data.
3. Every standard identity carries 6 atlas PNGs at 1280x1280 plus 150 frame PNGs at 256x256.
4. Average useful alpha area inside RTP frames is low. Across all inspected packages the average bbox fill is about 10.8%; human characters are typically ~14-22%. This means much of each 256x256 frame is transparent padding.
5. PNGs are already compressed; wrapping them in zip gives almost no additional compression. Root compressed and uncompressed sizes are nearly identical.
6. Loading thousands of image files would be bad for PWA startup, service-worker caching, memory, and update churn.

## Quality Comparison

RTP characters are visually higher fidelity than current runtime sprites:

- RTP standard frame: 256x256.
- Human character useful bbox example range: roughly 70-90 px wide by 125-172 px tall.
- Current PERLA1 object sprites: usually 64x64, compact, median useful bbox ~53x50 px.

This means RTP is appropriate for main/NPC characters and close-range event staging, but should not be treated like the current environment sprite set. Character assets need a separate loading, LOD, and manifest layer.

## Recommended Runtime Shape

Do not put RTP characters into the current `ASSET_MANIFEST` global preload. Create a separate RTP layer:

```text
assets/rtp/characters/
  portraits/main/<id>.webp
  sprites/main/<id>/<animation>.webp
  sprites/npc/<id>/<animation>.webp
  sprites/animals/<id>/<animation>.webp
data/rtp/
  rtp.characters.json
```

## Import Pipeline Proposal

Use a build-time importer, not runtime zip parsing.

Importer responsibilities:

- Read `RTP CHARACTERS.zip` from a source-only path.
- Validate paths to prevent zip-slip and reject unexpected extensions.
- Normalize identities to stable slugs.
- Verify dimensions, frame counts, atlas metadata, and alpha channels.
- Import only selected IDs for the current milestone.
- Import optimized main-character portraits as first-class RTP dialogue assets.
- Keep only atlas representation, not both atlas and frames.
- Convert portraits to WebP 512px or 384px.
- Convert sprite atlases to WebP where acceptable; keep PNG only if alpha edge quality visibly regresses.
- Emit `rtp.characters.json` with source identity, role, animations, facings, frame counts, generated asset paths, and source hash.
- Keep source zip out of Git and out of the shipped PWA.

## PWA Loading Policy

Recommended policy:

- Core game shell: preload current small `assets/raycast/` only.
- Main portraits: lazy load on first dialogue open, or prefetch shortly before a scripted RPG Maker-style dialogue when the event system can predict it.
- Character sprite atlases: lazy load by event/zone, with a small LRU texture cache.
- Animals: load per ambience zone, not globally.
- Service worker: do not precache full RTP by default. Use runtime cache with versioned URLs and a size cap.
- Debug counter: track loaded RTP identities, active atlas count, approximate decoded texture memory, and cache evictions.

## Security And Stability Rules

- Never ship or load the source nested zip directly in the browser.
- Never allow arbitrary zip extraction paths.
- Do not trust file extensions; verify magic bytes and decoded dimensions.
- Do not import NPC portraits by default while the dialogue portrait contract is main-character-only.
- Do not add RTP assets to the existing global `ASSET_MANIFEST`; it would regress startup and memory.
- Keep left-facing variants derived from right-facing sources unless measured runtime mirroring is too expensive on target devices.
- Add an asset budget gate before import: max package count, max decoded texture memory, max PWA cache bytes.
